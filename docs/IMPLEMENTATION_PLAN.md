# 《时光回响》实施计划

## 0. 计划定位

本计划只负责三件事：

1. 定义当前应该做什么
2. 定义谁负责什么
3. 约束后续模型不要跳步、不要重排顺序

文档优先级如下：

1. `IMPLEMENTATION_PLAN.md`
2. `CLAUDE.md`
3. `INTERVIEW_ENGINE_RULES.md`
4. `TIME_ECHO_PRD.md`
5. `QUESTION_BANK_HANDOFF.md`

---

## 1. 当前共识

当前版本已经确定以下规则：

1. 默认产物是约 `2000` 字的人物小传
2. 采用四段式闭环系统：基础题库 → 扩展题库 → 生成回忆录 → 补全机制
3. 采用分层生成机制：人生小传（2000 字）→ 人生故事（3000-5000 字）→ 人生回忆录（5000+ 字）
4. 采用双触发升级机制：主动触发 + 被动触发
5. `40` 滴墨水购买一轮基础访谈资格，不对应固定 `40` 题
6. 基础访谈默认上限为 `50` 个题位，但内容够了可提前结束
7. 若达到 `50` 题位仍不够，再由用户决定是否开启 `10` 滴 / 包、每包最多 `10` 题的扩展题目包
8. 题库必须坚持"选择题为主，自由补充为辅"
9. 深挖题可以很多，但不应默认要求老人打字
10. 每道正式题必须独立承担一个内容槽位，不能只是过渡题

## 2. 职责边界

### 2.1 Codex 的职责

Codex 负责：

- 正式题库本身
- 题库逻辑
- 题型结构
- 题库字段语义
- 题库文档与交接约束

正式题库唯一源头：

- `src/data/question-templates.ts`

### 2.2 Haiku 的职责

Haiku 负责：

- 题库消费层
- 调度器接线
- `/interview` 页面接线
- 预览主链
- 分流逻辑
- 生成与后续扩写实现

### 2.3 强约束

若无用户明确指示：

- Haiku 不重建题库
- Haiku 不重释题库字段
- Haiku 只消费正式题库
- Codex 不替 Haiku 实现主链页面、生成器、数据库结构

- 约 40 题左右，系统判断内容足够

- 问题包补全：AI 生成 3-5 题补全包

- 主动触发：用户点击"生成更完整版本"
- 人生故事生成：3000-5000 字
